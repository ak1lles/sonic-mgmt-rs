---
title: Docker Test Runner
description: Build and run integration tests inside a containerized test runner
---

The `docker/` directory provides a containerized test runner. The container compiles `sonic-mgmt` in a multi-stage build and packages it with SSH and network tools. Tests run inside the container and reach out to real SONiC devices on the network.

## Prerequisites

- Docker and Docker Compose v2
- cargo-make (`cargo install cargo-make`)
- A reachable SONiC testbed with SSH access

## Quick Start

```sh
# Build the test runner image
makers docker-build

# Open a shell inside the runner
makers docker-shell

# Run integration tests against a testbed
SONIC_TESTBED=my-testbed makers docker-test
```

## Image Layout

The Dockerfile uses a two-stage build.

**Builder stage** compiles the `sonic-mgmt` binary from source using `rust:1.75-bookworm`.

**Runtime stage** copies the binary into `debian:bookworm-slim` alongside `openssh-client`, `iputils-ping`, `iproute2`, `curl`, and `jq`.

The entrypoint is `sonic-mgmt`.

## Docker Compose

`docker/docker-compose.yml` defines a single `sonic-mgmt` service.

```yaml
services:
  sonic-mgmt:
    build:
      context: ..
      dockerfile: docker/Dockerfile
    image: sonic-mgmt-runner:latest
    container_name: sonic-mgmt-runner
    network_mode: host
    volumes:
      - ../config:/workspace/config:ro
      - ~/.ssh:/root/.ssh:ro
      - ../output:/workspace/output
```

### Volume Mounts

| Mount | Purpose |
|---|---|
| `config/` | Testbed and inventory files with real IPs. Never baked into the image. |
| `~/.ssh/` | SSH keys for device access |
| `output/` | Test results persist after the container exits |

Default network mode is `host` so the runner can reach lab management networks without Docker NAT.

### Environment Variables

| Variable | Default | Purpose |
|---|---|---|
| `SONIC_CONFIG` | `config/sonic-mgmt.toml` | Path to the application config inside the container |
| `SONIC_TESTBED` | | Active testbed name |
| `SONIC_LOG` | `info` | Log level |
| `SONIC_NETWORK_MODE` | `host` | Docker network mode override |
| `SSH_KEY_DIR` | `~/.ssh` | Host path to SSH keys |
| `SONIC_OUTPUT_DIR` | `../output` | Host path for test output |

## CLI Subcommands

All subcommands are available through `sonic-mgmt docker`.

| Subcommand | Description |
|---|---|
| `build [--no-cache]` | Build the test runner image |
| `up` | Start the container in detached mode |
| `down` | Stop and remove the container |
| `status` | Show container state, image info, and mounted volumes |
| `shell` | Open a bash shell inside the runner |
| `test [--testbed NAME] [-- ARGS...]` | Run integration tests inside the container |

## cargo-make Tasks

| Task | Description |
|---|---|
| `makers docker-build` | Build the test runner image |
| `makers docker-up` | Start the container |
| `makers docker-down` | Stop and remove the container |
| `makers docker-status` | Show container status |
| `makers docker-shell` | Open a shell in the runner |
| `makers docker-test` | Run integration tests in the container |
| `makers integration-test` | Run integration tests directly on the host |

## Integration Tests

Integration tests live in `crates/sonic-integration-tests/tests/integration.rs`. They are marked `#[ignore]` and skipped by default during `cargo test`.

### Target Resolution

Tests resolve the target device using the first matching source:

1. `SONIC_SSH_HOST` environment variable set -- use `SONIC_SSH_*` env vars directly
2. `SONIC_TESTBED` environment variable set -- load testbed config, use the primary DUT
3. Fall back to `127.0.0.1:22` with `admin`/`password`

### Running Tests

From inside the container:

```sh
SONIC_TESTBED=my-testbed makers docker-test
```

Directly on the host:

```sh
SONIC_TESTBED=my-testbed makers integration-test
```

With explicit env vars:

```sh
SONIC_SSH_HOST=10.0.0.1 SONIC_SSH_USER=admin SONIC_SSH_PASS=secret \
    cargo test -p sonic-integration-tests --test integration -- --ignored --test-threads=1
```

## CI Usage

```yaml
- run: cargo test
- run: docker compose -f docker/docker-compose.yml build
- run: |
    docker compose -f docker/docker-compose.yml run --rm \
      -e SONIC_TESTBED=ci-testbed sonic-mgmt \
      cargo test -p sonic-integration-tests --test integration -- --ignored
```

## Troubleshooting

### Image fails to build

```sh
# Check Docker daemon
docker info

# Build with no cache
sonic-mgmt docker build --no-cache
```

### Container cannot reach devices

Verify the container has network access to the management subnet.

```sh
# Check network mode
docker inspect sonic-mgmt-runner --format '{{.HostConfig.NetworkMode}}'

# Test connectivity from inside
docker compose -f docker/docker-compose.yml run --rm sonic-mgmt bash -c 'ping -c 1 <DUT_IP>'
```

### Tests fail with connection refused

Confirm the DUT is reachable and credentials are correct.

```sh
ssh -o ConnectTimeout=5 admin@<DUT_IP>
```
