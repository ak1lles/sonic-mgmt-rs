title: Device Operations

sonic-mgmt-rs provides a unified interface for operating on different device types. The framework handles connection management, command syntax differences, and output parsing automatically.

## Supported Device Types

| Device | Type Key | Connection | Capabilities |
|---|---|---|---|
| SONiC DUT | `sonic` | SSH, Telnet, Console | Full management: config, reboot, facts, CRM, queues |
| Arista EOS | `eos` | SSH | Enable mode, configure terminal, BGP, interfaces |
| Cisco IOS/NX-OS | `cisco` | SSH, Telnet | Configure terminal, show commands, BGP |
| Fanout | `fanout` | SSH | Port speed, port shutdown/enable, VLAN operations |
| PTF | `ptf` | SSH | Run PTF tests, file transfer, IP management |

## Listing Devices

```sh
# List all devices from inventory
sonic-mgmt device list

# Show details for a specific device
sonic-mgmt device show sonic-switch-1
```

## Executing Commands

Run arbitrary commands on a device:

```sh
# Single command
sonic-mgmt device exec sonic-switch-1 "show version"

# Multiple commands
sonic-mgmt device exec sonic-switch-1 "show ip bgp summary"
sonic-mgmt device exec sonic-switch-1 "show interfaces status"
```

The framework automatically handles:
- SSH connection setup and authentication
- Connection pooling (reuses connections to the same host)
- Exit code, stdout, and stderr capture
- Command timeout enforcement

## Collecting Facts

Device facts are structured data collected by parsing device output. Facts are **cached** with a 5-minute TTL to avoid redundant collection.

```sh
sonic-mgmt device facts sonic-switch-1
```

### Fact Categories

**Basic Facts** -- parsed from `show version` and `show platform`:

| Fact | Example |
|---|---|
| `hostname` | `sonic-switch-1` |
| `hwsku` | `ACS-MSN2700` |
| `os_version` | `20230531.6` |
| `platform` | `x86_64-mlnx_msn2700-r0` |
| `serial` | `MLX1234567890` |
| `model` | `MSN2700` |
| `mac_address` | `b8:6a:97:xx:xx:xx` |
| `uptime_seconds` | `86400` |

**BGP Facts** -- parsed from `show ip bgp summary`:

| Fact | Description |
|---|---|
| `asn` | Local AS number |
| `neighbors` | List of BGP neighbors with state, AS, prefixes received |
| `routes` | BGP route table entries |

**Interface Facts** -- parsed from `show interfaces status`, `show vlan brief`, `show portchannel summary`:

| Fact | Description |
|---|---|
| `ports` | Physical port status (up/down, speed, MTU) |
| `vlans` | VLAN configuration with member interfaces |
| `lags` | LAG/PortChannel members |
| `loopbacks` | Loopback interface IPs |

**Config Facts** -- parsed from running configuration:

| Fact | Description |
|---|---|
| `running_config` | Complete running configuration |
| `features` | Enabled features and their status |
| `services` | SONiC service container status |

## Rebooting Devices

```sh
# Cold reboot (default)
sonic-mgmt device reboot sonic-switch-1

# Warm reboot
sonic-mgmt device reboot sonic-switch-1 --type warm

# Fast reboot
sonic-mgmt device reboot sonic-switch-1 --type fast
```

### Reboot Types

| Type | Description | Downtime |
|---|---|---|
| `cold` | Full system restart | Minutes |
| `warm` | Warm restart with minimal traffic loss | Seconds |
| `fast` | Fast restart with brief traffic loss | Seconds |
| `power-cycle` | Physical power cycle (via console server) | Minutes |
| `watchdog` | Trigger watchdog reboot | Minutes |
| `supervisor` | Supervisor restart (modular chassis) | Varies |
| `kdump` | Kernel crash dump and reboot | Minutes |

After reboot, the framework polls the device until it's ready (SSH accessible and critical services running).

## Connection Types

### SSH (Default)

Standard SSH with key-based or password authentication:

```toml
[devices.my-device.connection]
connection_type = "ssh"
port = 22
```

The SSH client:
- Tries key-based auth first, falls back to password
- Accepts all host keys (appropriate for lab environments)
- Supports configurable connect timeout (default 30s) and command timeout (default 120s)

### Telnet

For devices that only support telnet:

```toml
[devices.my-device.connection]
connection_type = "telnet"
port = 23
```

Handles RFC 854/855 telnet protocol including IAC option negotiation.

### Console

Out-of-band serial console access via conserver:

```toml
[devices.my-device.console]
server   = "console-server-1"
port     = 3001
protocol = "ssh"
```

Used for:
- Recovery from unresponsive devices
- BIOS-level operations
- Devices unreachable via management network

## SONiC-Specific Operations

For SONiC DUTs, additional operations are available:

```sh
# Reload configuration
sonic-mgmt device exec sonic-switch-1 "config reload -y"

# Check container status
sonic-mgmt device exec sonic-switch-1 "docker ps --format 'table {{.Names}}\t{{.Status}}'"

# Check CRM resources
sonic-mgmt device exec sonic-switch-1 "crm show resources all"

# Check queue counters
sonic-mgmt device exec sonic-switch-1 "show queue counters"
```

## Connection Pooling

The framework automatically pools and reuses connections. When you run multiple commands against the same device, only one SSH connection is established. Connections are keyed by `(host, port, connection_type)` and reused until they become idle.
