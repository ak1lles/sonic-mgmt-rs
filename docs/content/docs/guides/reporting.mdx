sonic-mgmt-rs provides tools for parsing test results, tracking test coverage, and uploading results to Azure Data Explorer (Kusto) for centralized analytics.

## Parsing JUnit XML

The framework outputs test results in JUnit XML format by default. Parse results with:

```sh
sonic-mgmt report parse output/results.xml
```

This displays:
- Test suite name and total counts (pass/fail/skip/error)
- Per-test case results with duration
- Failure messages and stack traces
- System output and error logs

### JUnit XML Structure

The parser handles standard JUnit XML with these elements:
- `<testsuite>` -- suite-level metadata and counts
- `<testcase>` -- individual test with classname, name, time
- `<failure>` / `<error>` / `<skipped>` -- outcome details
- `<system-out>` / `<system-err>` -- captured output

## Uploading to Kusto

Upload test results to Azure Data Explorer for centralized tracking:

```sh
sonic-mgmt report upload output/results.xml
```

### Configuration

Configure the Kusto backend in `sonic-mgmt.toml`:

```toml
[reporting]
backend_url  = "https://your-cluster.kusto.windows.net"
auth_method  = "azure_default"
database     = "sonic_test_results"
table        = "test_runs"
```

### Authentication Methods

| Method | Description |
|---|---|
| `azure_default` | Azure Default Credential chain (recommended) |
| `azure_cli` | Uses `az` CLI authentication |
| `device_code` | Interactive device code flow |
| `managed_identity` | Azure Managed Identity (for VMs/containers) |
| `app_key` | Application ID + secret |
| `user_token` | User-provided access token |
| `app_token` | Application-provided token |

### Upload Metadata

Each upload includes metadata:

| Field | Description |
|---|---|
| `testbed` | Name of the testbed used |
| `timestamp` | When the test run occurred |
| `uploader` | Who initiated the upload |
| `topology` | Topology type used |
| `platform` | Platform tested |

### Retry Logic

Uploads automatically retry on transient failures with backoff.

## Coverage Tracking

Track SAI (Switch Abstraction Interface) and SWSS (Switch State Service) test coverage:

```sh
sonic-mgmt report coverage
```

This analyzes test results to determine which SAI APIs and SWSS features have been exercised by the test suite.

## Local Storage

Results can also be stored locally for later analysis:

```sh
# Results are written to the output directory
sonic-mgmt test run --path tests/ --output output/

# View stored results
ls output/
# results.xml
# results.json
# test_artifacts/
```

## Output Formats

Configure the output format in `sonic-mgmt.toml` or via CLI:

| Format | File | Best For |
|---|---|---|
| `junit_xml` | `results.xml` | CI/CD integration, Jenkins, Azure DevOps |
| `json` | `results.json` | Programmatic analysis |
| `toml` | `results.toml` | Human-readable, config-like |
| `csv` | `results.csv` | Spreadsheet import |
| `html` | `results.html` | Viewing in a browser |
