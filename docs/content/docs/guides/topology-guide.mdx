sonic-mgmt-rs can auto-generate 13 different network topologies. Each topology defines the VMs, VLANs, host interfaces, LAG links, and IP addressing needed to test a SONiC switch in a specific role.

## Topology Overview

### Leaf Topologies (T0 family)

Used when testing a **leaf switch** connected to servers and uplink spine switches.

| Topology | VMs | Host Interfaces | Description |
|---|---|---|---|
| `t0` | 4 | 32 | Standard leaf with 4 spine neighbors and 32 server ports |
| `t0-64` | 64 | 64 | Scaled leaf with 64 neighbors |
| `t0-116` | 116 | 116 | Large leaf deployment |

### Specialty Topologies

| Topology | VMs | Host Interfaces | Description |
|---|---|---|---|
| `dualtor` | 4 | 32 | Dual top-of-rack with active-standby redundancy |
| `mgmt-tor` | 4 | 32 | Management TOR topology |
| `m0-vlan` | 4 | 32 | M0 topology with VLAN segmentation |

### PTF-Only Topologies

No VMs, just direct PTF packet injection:

| Topology | Host Interfaces | Description |
|---|---|---|
| `ptf-32` | 32 | PTF with 32 ports |
| `ptf-64` | 64 | PTF with 64 ports |
| `ptf` | 8 | Minimal PTF |

These are the simplest to set up -- they require no VM images and test pure forwarding behavior using crafted packets.

## Generating Topologies

### List Available Topologies

```sh
sonic-mgmt topology list
```

### View Topology Details

```sh
sonic-mgmt topology show t0
```

Shows: number of VMs, VLANs, host interfaces, LAG links, and IP allocation scheme.

### Generate a Topology Definition

```sh
sonic-mgmt topology generate t0
```

Outputs a complete `TopologyDefinition` including:
- VM instances with names (ARISTA01T1, ARISTA02T1, etc.) and management IPs
- VLAN definitions with IDs, names, and prefixes
- Host interface mappings (DUT port -> PTF port -> VLAN)
- LAG link definitions (for LAG topologies)
- IP pairs (IPv4 and IPv6) for each connection

### Render Configuration Files

```sh
# Generate minigraph XML
sonic-mgmt topology render t0 --format minigraph

# Generate config_db.json
sonic-mgmt topology render t0 --format config_db

# Generate inventory file
sonic-mgmt topology render t0 --format inventory
```

## IP Address Allocation

Topologies use deterministic IP allocation:

| Network | Base Address | Purpose |
|---|---|---|
| IPv4 links | `10.0.0.0/16` | Point-to-point links between DUT and neighbors |
| IPv6 links | `fc00::/96` | IPv6 point-to-point links |
| VM management | `10.250.0.2` | Starting management IP for neighbor VMs |
| VLANs | `1000` | Starting VLAN ID |

Each topology allocates IPs from these ranges without overlap, allowing multiple topologies to coexist on the same server.

## VM Types

Neighbor VMs can be different virtual switch types:

| VM Type | Key | Description |
|---|---|---|
| Arista vEOS | `veos` | Most commonly used, good EOS compatibility |
| Arista cEOS | `ceos` | Containerized EOS, lighter weight |
| Virtual SONiC | `vsonic` | SONiC running as a VM |
| Virtual Cisco | `vcisco` | Cisco IOS/NX-OS VM |
| Containerized SONiC | `csonic` | SONiC running in a container |

## Choosing a Topology

| Testing Goal | Recommended | Why |
|---|---|---|
| General L2/L3 forwarding | `t0` | Standard leaf, broadest test coverage |
| BGP at scale | `t1-64` | 64 BGP neighbors |
| LAG functionality | `t1-lag` | LAG links for LACP testing |
| Dual-TOR failover | `dualtor` | Tests active-standby behavior |
| Quick packet tests | `ptf` | No VMs, fast setup |
| Full fabric depth | `t2` | 3-tier topology |
| VLAN features | `m0-vlan` | VLAN segmentation testing |
