The `sonic-mgmt` binary is the single entry point for all operations. Every command supports `--help` for inline documentation.

## Global Options

```sh
sonic-mgmt [OPTIONS] <COMMAND>
```

| Option | Env Var | Default | Description |
|---|---|---|---|
| `-c, --config <PATH>` | `SONIC_CONFIG` | `sonic-mgmt.toml` | Path to the main configuration file |
| `--log-level <LEVEL>` | `SONIC_LOG` | `info` | Log level: `trace`, `debug`, `info`, `warn`, `error` |

## Commands

### init

Run the first-time setup wizard to create `sonic-mgmt.toml`:

```sh
sonic-mgmt init
```

Interactively prompts for connection defaults, testing settings, and logging preferences.

### testbed

Manage testbed lifecycle.

```sh
sonic-mgmt testbed <SUBCOMMAND>
```

| Subcommand | Description |
|---|---|
| `list` | List all defined testbeds |
| `show <name>` | Show detailed info about a testbed |
| `deploy <name>` | Deploy topology to a testbed |
| `teardown <name>` | Tear down a deployed testbed |
| `health <name>` | Run health checks on all devices |
| `refresh <name>` | Refresh testbed configuration |
| `upgrade <name>` | Upgrade SONiC image on DUTs |
| `wizard` | Interactive testbed creation wizard |

**Examples:**

```sh
# Deploy the active testbed
sonic-mgmt testbed deploy

# Deploy a specific testbed
sonic-mgmt testbed deploy vms-t0

# Check testbed health
sonic-mgmt testbed health vms-t0

# Tear down and clean up
sonic-mgmt testbed teardown vms-t0
```

### device

Interact with individual network devices.

```sh
sonic-mgmt device <SUBCOMMAND>
```

| Subcommand | Description |
|---|---|
| `list` | List all devices in the inventory |
| `show <hostname>` | Show device details and status |
| `connect <hostname>` | Open an interactive SSH session |
| `exec <hostname> <command>` | Execute a command on a device |
| `reboot <hostname>` | Reboot a device (cold, warm, fast) |
| `facts <hostname>` | Collect and display device facts |
| `wizard` | Interactive device addition wizard |

**Examples:**

```sh
# List all devices
sonic-mgmt device list

# Execute a command on a DUT
sonic-mgmt device exec sonic-switch-1 "show version"

# Collect device facts
sonic-mgmt device facts sonic-switch-1

# Reboot with warm reboot
sonic-mgmt device reboot sonic-switch-1 --type warm
```

### topology

Generate and inspect network topologies.

```sh
sonic-mgmt topology <SUBCOMMAND>
```

| Subcommand | Description |
|---|---|
| `list` | List all supported topology types |
| `show <type>` | Show topology details (VMs, VLANs, interfaces) |
| `generate <type>` | Generate a complete topology definition |
| `render <type>` | Render topology to minigraph XML / config_db.json |

**Examples:**

```sh
# List all 13 supported topologies
sonic-mgmt topology list

# Show T0 topology details
sonic-mgmt topology show t0

# Generate T1-LAG topology definition
sonic-mgmt topology generate t1-lag

# Render minigraph XML for a topology
sonic-mgmt topology render t0 --format minigraph
```

### test

Discover and execute tests.

```sh
sonic-mgmt test <SUBCOMMAND>
```

| Subcommand | Description |
|---|---|
| `discover` | Find tests matching filters |
| `run` | Execute tests |
| `wizard` | Interactive test planning wizard |
| `results` | View test results |

**Run options:**

| Option | Default | Description |
|---|---|---|
| `--path <DIR>` | `.` | Directory to scan for test files |
| `--pattern <GLOB>` | `test_*.toml` | File pattern for test discovery |
| `--tag <TAG>` | | Filter by tag (repeatable) |
| `--topology <TYPE>` | | Filter by topology type |
| `--platform <PLATFORM>` | | Filter by platform |
| `-j, --jobs <N>` | `1` | Number of parallel workers |
| `--timeout <SECS>` | `900` | Per-test timeout |
| `--fail-fast` | `false` | Stop on first failure |
| `--output <DIR>` | `output` | Output directory for artifacts |
| `--dry-run` | `false` | Discover without executing |

**Examples:**

```sh
# Discover all tests
sonic-mgmt test discover --path tests/

# Run BGP tests with 4 workers
sonic-mgmt test run --path tests/ --tag bgp -j 4

# Run T0 tests with fail-fast
sonic-mgmt test run --path tests/ --topology t0 --fail-fast

# Dry run to see what would execute
sonic-mgmt test run --path tests/ --dry-run
```

### report

Parse and upload test reports.

```sh
sonic-mgmt report <SUBCOMMAND>
```

| Subcommand | Description |
|---|---|
| `parse <FILE>` | Parse a JUnit XML report and display results |
| `upload <FILE>` | Upload results to Kusto (Azure Data Explorer) |
| `coverage` | View SAI/SWSS test coverage metrics |

**Examples:**

```sh
# Parse JUnit XML
sonic-mgmt report parse output/results.xml

sonic-mgmt report upload output/results.xml

# View coverage report
sonic-mgmt report coverage
```

### sdn

SDN protocol operations via gRPC.

```sh
sonic-mgmt sdn <SUBCOMMAND>
```

| Subcommand | Description |
|---|---|
| `gnmi get <path>` | gNMI Get request |
| `gnmi set <path> <value>` | gNMI Set request |
| `gnmi subscribe <path>` | gNMI Subscribe (streaming) |
| `gnoi reboot <target>` | gNOI Reboot operation |
| `gnoi ping <target>` | gNOI Ping operation |
| `p4rt write <table> <entry>` | P4Runtime table write |

**Examples:**

```sh
# Get interface status via gNMI
sonic-mgmt sdn gnmi get /interfaces/interface[name=Ethernet0]/state/oper-status

# Subscribe to interface counters
sonic-mgmt sdn gnmi subscribe /interfaces/interface/state/counters

# Reboot via gNOI
sonic-mgmt sdn gnoi reboot sonic-switch-1
```

### docker

Manage the containerized test runner.

```sh
sonic-mgmt docker <SUBCOMMAND>
```

| Subcommand | Description |
|---|---|
| `build [--no-cache]` | Build the test runner image |
| `up` | Start the container in detached mode |
| `down` | Stop and remove the container |
| `status` | Show container state and image info |
| `shell` | Open a shell inside the runner |
| `test [--testbed NAME]` | Run integration tests in the container |

**Examples:**

```sh
# Build the test runner image
sonic-mgmt docker build

# Run integration tests against a testbed
sonic-mgmt docker test --testbed my-testbed

# Open a shell for manual exploration
sonic-mgmt docker shell

# Check container status
sonic-mgmt docker status
```
